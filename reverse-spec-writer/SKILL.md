# Reverse Spec Writer

Reads a codebase and produces the spec.md that should have been written before it was built — documenting intent, architecture, data models, and API contracts as they actually exist.

**Input**: Project codebase
**Output**: spec.md that describes what was built

## Trigger phrases
- "document this codebase"
- "reverse engineer spec"
- "write architecture doc"
- "I inherited this code"
- "understand existing system"
- "generate spec from code"

## Step-by-step workflow

### STEP 1 — Read the codebase

Scan the project root. Identify:
- Entry points (`main.go`, `cmd/`, `src/index.ts`, `app.py`, etc.)
- Directory structure — what each top-level folder is for
- Config files (`docker-compose.yml`, `.env.example`, `Makefile`, `package.json`)
- README if present (use as a hint, not as truth)
- Key data files (schema files, `.proto`, `schema.graphql`, migration files)

Do NOT trust README content — verify everything against the actual code.

---

### STEP 2 — Map the architecture

Identify the system's layers and how they connect:

```
[Transport layer] → [Business logic layer] → [Data layer]
     (HTTP/gRPC)          (service/use-case)      (DB/cache/queue)
```

For each layer, list:
- The files/packages that implement it
- What it does
- What it depends on

Identify external dependencies:
- Databases (Postgres, Redis, MongoDB, etc.)
- Message queues (Kafka, RabbitMQ, SQS)
- External APIs called
- Third-party services

---

### STEP 3 — Extract data models

Find all domain entities:
- Structs / interfaces / classes / types
- Database schema (migration files, ORM models)
- Proto messages (if gRPC)
- API request/response shapes

For each entity, document:
- Fields and types
- Relationships (foreign keys, embedded types)
- Validation rules (if present in code)

---

### STEP 4 — Document API contracts

For each exposed endpoint or RPC method:
- Method name and HTTP verb + path (or gRPC service/method)
- Request shape
- Response shape
- Auth requirement (if detectable)
- Error cases (if documented in code)

---

### STEP 5 — Identify key behaviours and business rules

Read the service/use-case layer and extract:
- What the system is designed to do in plain English
- Key business rules (e.g. "a listing cannot be published without a price")
- State machines (if present)
- Background jobs and scheduled tasks

---

### STEP 6 — Write the spec.md

Produce a structured `spec.md`:

```markdown
# [System Name] — Specification

> Auto-generated by reverse-spec-writer. Reflects code state as of [date].

## Overview
[2–3 sentences: what this system does and why it exists]

## Architecture
[Diagram in ASCII or description of layers]

## Tech Stack
| Layer | Technology |
|-------|------------|
| Language | Go 1.22 |
| Transport | gRPC + HTTP (grpc-gateway) |
| Database | PostgreSQL (sqlx) |
| Cache | Redis |
| Queue | Kafka |

## Data Models
### [EntityName]
| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | string (UUID) | yes | Primary key |
| ... | ... | ... | ... |

## API / RPC Contracts
### [ServiceName]
| Method | Input | Output | Auth |
|--------|-------|--------|------|
| Create | CreateRequest | Entity | JWT |
| ... | ... | ... | ... |

## Business Rules
- [Rule 1]
- [Rule 2]

## Key Flows
### [Flow name]
1. [Step 1]
2. [Step 2]

## External Dependencies
| Service | Purpose | Config key |
|---------|---------|------------|
| Postgres | Primary store | DATABASE_URL |

## Background Jobs
| Job | Schedule | Purpose |
|-----|----------|----------|
| outbox-worker | continuous | Publishes outbox events to Kafka |

## Known Gaps / TODOs
[List anything in the code marked TODO, FIXME, or that is obviously incomplete]
```

---

### STEP 7 — Output

Write to `spec.md` in the project root. Present the file.

Print summary:
```
✅ spec.md generated for [system name]
  Entities documented: N
  API methods documented: N
  Business rules extracted: N
  ⚠️ Gaps found: N (see Known Gaps section)
```

---

## Output format spec

- Single `spec.md` file at project root
- Markdown tables for data models and API contracts
- ASCII diagram for architecture (or description if too complex)
- "Auto-generated" disclaimer at top

---

## Quality checklist

- [ ] Architecture section describes all layers
- [ ] All top-level packages/folders are accounted for
- [ ] Every entity has its fields documented
- [ ] Every API method has input/output documented
- [ ] External dependencies listed with config keys
- [ ] Known Gaps section present (even if empty)
- [ ] No speculation — only what the code actually does

---

## Edge cases

- **README contradicts code**: Note the discrepancy in Known Gaps — "README claims X but code does Y."
- **No clear layering**: Document as flat structure; note that separation of concerns may be a gap.
- **Monorepo**: Produce one spec.md per service/package, not one giant file.
- **Generated code** (proto, sqlc, wire): Skip generated files; document the source (`.proto`, SQL queries) instead.
- **Very large codebase**: Focus on the entry points, key service interfaces, and data models; mark as partial scan.
